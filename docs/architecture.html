Ethy Ads01

Problem

Telegram ads still happen like side deals.

1

No standard flow

Advertisers hunt for channel owners, DM them, negotiate manually, and hope the person on the other side is real.

2

Trust breaks payments

The advertiser worries the post will be deleted. The publisher worries they will not get paid after the ad goes live.

3

Discovery is global, ads are not

Language, usernames, informal payment rails, and changing identities make cross-border channel ads painful.

The pain is not creating an ad. It is proving the deal was honored.02

Product loop

A simple booking flow with escrow-backed incentives.

1

Create

Advertiser describes the ad, budget, target channel, and duration through the bot or client.

2

Lock

Funds move from available balance into a contract-backed campaign escrow.

3

Offer

Publisher receives exact content, payment terms, and accept or reject actions in Telegram.

4

Post + verify

After acceptance, the bot publishes the approved ad and the server verifies channel, text, image hash, and timing.

5

Settle

If the post stays valid, the publisher gets paid. If not, the advertiser is refunded.

Core guarantee: pay only for verified delivery.03

What we built

One system, three trust layers.

Telegram interface Onboarding, channel proof posting, campaign offers, accept/reject, bot posting, verification, and status messages.
React client Public proof pages, campaign visibility, ENS records access, and wallet-oriented dashboard pieces.
AI helpers Intake, copy suggestions, safety review, offer generation, negotiation, translation-ready roles.

TypeScript server

The server is the orchestrator: Express API, Telegram bot handlers, campaign state machine, verification logic, persistence, agent gateway, Dynamic wallet gateway, and Viem contract calls.

DRAFT → FUNDED → OFFERED → ACCEPTED → AWAITING_POST → ACTIVE → COMPLETED / REFUNDED
Solidity escrow ERC-20 deposit, internal balances, campaign locking, verifier start, completion, refund, and withdrawal.
ENS proof layer Users, AI agents, ads, and lifecycle events become readable ENS records.
CCIP-read resolver Dynamic off-chain ad records resolve through a standard EIP-3668 gateway.
No AI finalizes money. Deterministic verification and contract state do.04

Wallet and contract layer

Dynamic wallets power the dev Telegram flow.

Dynamic

We used Dynamic's Telegram bot starter as a foundation, then expanded it into Ethy Ads. In the backend, Dynamic server wallets are created with two-of-two threshold signing and are used to sign messages, typed data, and prepared transactions.

Template attribution: github.com/dynamic-labs-oss/tg-bot-starter

Escrow contract

The Solidity contract tracks available balances by wallet and token, locks advertiser funds into campaigns, and lets an authorized verifier start, complete, or refund a campaign.

The contract stores money-critical facts, while Telegram content and proof metadata stay off-chain.

Dynamic wallet → approve / deposit / create campaign → AdEscrow.sol → start / complete / refund → ENS lifecycle proof
Wallet execution is separated from AI suggestions and campaign copy.07
